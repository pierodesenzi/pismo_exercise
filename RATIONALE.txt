RATIONALE
=========

Este documento descreve as decisões de projeto e arquitetura adotadas no sistema de transações.

1. Estrutura do projeto
-----------------------

O projeto segue a arquitetura de camadas comum do Spring Boot:

- **controller**
  - `AccountController` (CRUD de contas)
  - `TransactionController` (criação/consulta de transações)
  - `HealthController`

- **service**
  - `AccountService`, `TransactionService`

- **repository**
  - `AccountRepository`, `TransactionRepository`

- **model**
  - `Account`, `Transaction` (OperationType logic agora é interno na Service)

Motivos principais:

- separação de responsabilidades;
- testabilidade e manutenção;
- uso de Spring Data JPA para reduzir boilerplate de persistência.

2. Endpoints implementados
-------------------------

- `GET /health`: status de saúde (UP) para readiness/liveness.

Contas:
- `POST /accounts`: cria conta (`document_number`).
- `GET /accounts/{accountId}`: consulta conta por id.
- `GET /accounts`: lista todas as contas.

Transações:
- `POST /transactions`: cria transação com validações (conta/existente, tipo operação/existente, ajuste de valor e data).  
- `GET /transactions/account/{accountId}`: lista transações de uma conta.


4. Regras de negócio (TransactionService)
-----------------------------------------

- Verifica se `account_id` existe.
- Verifica se `operation_type_id` existe.
- Ajuste de sinal de valor com base em tipo de operação:
  - 1 (COMPRA A VISTA), 2 (COMPRA PARCELADA), 3 (SAQUE): `amount` negativo.
  - 4 (PAGAMENTO): `amount` positivo.
- Define `event_date` para `LocalDateTime.now()`.

5. Persistência e Docker
-----------------------

- `application.properties` aponta para PostgreSQL em container:
  - `jdbc:postgresql://db:5432/transactiondb`
  - `spring.jpa.hibernate.ddl-auto=update`

- `docker-compose.yml` define serviços:
  - `app`: aplicação Spring Boot.
  - `db`: PostgreSQL 15 com volume persistido.

A estratégia foi garantir que ambiente seja reprodutível e fácil de subir (`docker-compose up --build`).

6. Testes adicionados
--------------------

- `AccountControllerTest`
  - criação de conta
  - consulta por ID
  - listagem de contas

- `TransactionControllerTest`
  - criação de transação válida
  - consulta de transações por conta
  - erro quando conta não existe

- Dependências de teste: `spring-boot-starter-test` (JUnit5), `h2` (banco in-memory)
- Configuração: `src/test/resources/application.properties` com H2 para isolamento de testes
- Plugin: `maven-surefire-plugin` para detecção correta de testes JUnit5


Observação final
---------------

A API agora cobriu requisitos exigidos:
- todos os endpoints de transação/conta presentes;
- endpoint de tipos de operação removido (não necessário);
- persistência e Docker configurados;
- testes automatizados para fluxo crítico.
